# Data Model: Multi-Brand Design Token Platform

**Feature**: `001-multibrand-tokens`
**Date**: 2026-03-02

## D1 Schema (SQLite via Cloudflare D1)

### Table: `brands`

Source of truth for every registered brand. Stores seed color as three discrete columns (not JSON)
to support indexed queries and avoid parsing overhead. Pre-computed color scale stored inline as
JSON.

```sql
CREATE TABLE IF NOT EXISTS brands (
  id            TEXT    PRIMARY KEY,          -- URL-safe slug, e.g. "acme"
  display_name  TEXT    NOT NULL,             -- Human-readable, e.g. "Acme Corp"
  seed_l        REAL    NOT NULL,             -- OKLCH L channel: [0, 1]
  seed_c        REAL    NOT NULL,             -- OKLCH C channel: [0, ~0.37]
  seed_h        REAL    NOT NULL,             -- OKLCH H channel: [0, 360)
  font_display  TEXT    NOT NULL,             -- Display/hero font family (required)
  font_heading  TEXT    NOT NULL,             -- Heading font family (required)
  font_body     TEXT    NOT NULL,             -- Body font family (required)
  primitives    TEXT,                         -- JSON: non-color overrides (spacing, radius, etc.)
  color_scale   TEXT    NOT NULL DEFAULT '{}',-- JSON: pre-computed light+dark OKLCH scale
  valid_status  TEXT    NOT NULL DEFAULT 'pending', -- 'valid' | 'invalid' | 'pending'
  created_at    TEXT    NOT NULL,
  updated_at    TEXT    NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_brands_valid_status ON brands (valid_status);
```

**Validation rules:**
- `id`: URL-safe characters only (`[a-z0-9-]`), unique, 1–64 chars
- `seed_l`: 0.0 ≤ value ≤ 1.0
- `seed_c`: 0.0 ≤ value ≤ 0.4
- `seed_h`: 0.0 ≤ value < 360.0
- `font_display`, `font_heading`, `font_body`: non-empty strings (no platform default)
- `primitives`: valid JSON object or NULL
- `color_scale`: populated by the service layer on every write; never NULL after creation

**`primitives` JSON schema** (all keys optional; fall back to platform defaults when absent):
```json
{
  "spacing": {
    "base": "1rem",
    "scale": [0.25, 0.5, 0.75, 1, 1.5, 2, 3, 4, 6, 8]
  },
  "borderRadius": {
    "sm": "2px", "md": "4px", "lg": "8px", "xl": "16px", "full": "9999px"
  },
  "motion": {
    "durationFast": "150ms",
    "durationBase": "250ms",
    "durationSlow": "400ms",
    "easingStandard": "cubic-bezier(0.4, 0, 0.2, 1)",
    "easingDecelerate": "cubic-bezier(0, 0, 0.2, 1)",
    "easingAccelerate": "cubic-bezier(0.4, 0, 1, 1)"
  },
  "elevation": {
    "sm": "0 1px 2px rgba(0,0,0,0.05)",
    "md": "0 4px 6px rgba(0,0,0,0.07)",
    "lg": "0 10px 15px rgba(0,0,0,0.10)",
    "xl": "0 20px 25px rgba(0,0,0,0.15)"
  }
}
```

**`color_scale` JSON schema** (generated by `color/oklch.ts` at write time):

### Table: `semantic_tokens`

Platform-level definitions shared across all brands. Maps a semantic name to a color scale key for
each color scheme, plus an optional non-color primitive reference.

```sql
CREATE TABLE IF NOT EXISTS semantic_tokens (
  name              TEXT    PRIMARY KEY,      -- e.g. "color.surface.default"
  tier1_ref_light   TEXT,                    -- scale key for light mode, e.g. "50"
  tier1_ref_dark    TEXT,                    -- scale key for dark mode, e.g. "900"
  primitive_ref     TEXT,                    -- for non-color tokens, e.g. "spacing.base"
  description       TEXT,
  created_at        TEXT    NOT NULL,
  updated_at        TEXT    NOT NULL,
  -- Constraint: exactly one of (tier1_ref_light+tier1_ref_dark) or primitive_ref must be set
  CHECK (
    (tier1_ref_light IS NOT NULL AND tier1_ref_dark IS NOT NULL AND primitive_ref IS NULL) OR
    (tier1_ref_light IS NULL AND tier1_ref_dark IS NULL AND primitive_ref IS NOT NULL)
  )
);
```

**Validation rules:**
- `name`: dot-notation, `[a-z][a-z0-9.]*`, unique, max 128 chars
- Color semantic tokens: both `tier1_ref_light` and `tier1_ref_dark` must be set; must be valid
  scale keys (one of: `50 100 200 300 400 500 600 700 800 900 950`)
- Non-color semantic tokens: `primitive_ref` must be a dot-path that resolves in the brand's
  `primitives` JSON or the platform defaults

### Table: `component_tokens`

Platform-level default mapping: every UI component property → a semantic token.

```sql
CREATE TABLE IF NOT EXISTS component_tokens (
  name              TEXT    PRIMARY KEY,      -- e.g. "button.background.default"
  semantic_token    TEXT    NOT NULL,         -- FK → semantic_tokens.name
  description       TEXT,
  created_at        TEXT    NOT NULL,
  updated_at        TEXT    NOT NULL,
  FOREIGN KEY (semantic_token) REFERENCES semantic_tokens (name)
);

CREATE INDEX IF NOT EXISTS idx_component_tokens_semantic
  ON component_tokens (semantic_token);
```

**Validation rules:**
- `name`: dot-notation, `[a-z][a-z0-9.]*`, unique, max 128 chars
- `semantic_token`: must reference an existing row in `semantic_tokens` (FK enforced)

---

### Table: `brand_token_overrides`

Sparse per-brand overrides. Only rows that differ from the shared `component_tokens` default are
stored. A brand with no overrides has zero rows here.

```sql
CREATE TABLE IF NOT EXISTS brand_token_overrides (
  brand_id                TEXT    NOT NULL,   -- FK → brands.id
  component_token         TEXT    NOT NULL,   -- FK → component_tokens.name
  semantic_token_override TEXT    NOT NULL,   -- FK → semantic_tokens.name
  created_at              TEXT    NOT NULL,
  updated_at              TEXT    NOT NULL,
  PRIMARY KEY (brand_id, component_token),
  FOREIGN KEY (brand_id)                REFERENCES brands (id)            ON DELETE CASCADE,
  FOREIGN KEY (component_token)         REFERENCES component_tokens (name),
  FOREIGN KEY (semantic_token_override) REFERENCES semantic_tokens (name)
);

CREATE INDEX IF NOT EXISTS idx_overrides_brand ON brand_token_overrides (brand_id);
```

### Table: `token_validation_log`

Async validation results per brand. One row per validation run; the latest run is the current
status. `brands.valid_status` is the denormalized fast-path field; this table holds the details.

```sql
CREATE TABLE IF NOT EXISTS token_validation_log (
  id          TEXT    PRIMARY KEY,            -- UUID
  brand_id    TEXT    NOT NULL,
  run_at      TEXT    NOT NULL,               -- ISO 8601
  status      TEXT    NOT NULL,               -- 'pass' | 'fail'
  errors      TEXT,                           -- JSON array of ValidationError objects
  FOREIGN KEY (brand_id) REFERENCES brands (id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_validation_brand_run
  ON token_validation_log (brand_id, run_at DESC);
```

**`errors` JSON schema:**
```json
[
  {
    "code": "BROKEN_REF",
    "token": "button.background.default",
    "ref": "color.action.primary",
    "message": "Referenced semantic token does not exist"
  },
  {
    "code": "CONTRAST_FAIL",
    "foreground": "color.text.primary",
    "background": "color.surface.default",
    "scheme": "dark",
    "actual": 3.2,
    "required": 4.5,
    "message": "WCAG AA contrast failure in dark mode"
  },
  {
    "code": "CIRCULAR_REF",
    "cycle": ["button.bg", "color.action", "button.bg"],
    "message": "Circular reference detected"
  }
]
```

## Hot-Path Resolution Query

The critical SQL for serving a brand's token set — executed on every D1 cache miss:

```sql
SELECT
  ct.name                                                    AS component_token,
  COALESCE(bto.semantic_token_override, ct.semantic_token)   AS effective_semantic,
  st.tier1_ref_light,
  st.tier1_ref_dark,
  st.primitive_ref
FROM component_tokens ct
LEFT JOIN brand_token_overrides bto
  ON bto.brand_id = ?1 AND bto.component_token = ct.name
JOIN semantic_tokens st
  ON st.name = COALESCE(bto.semantic_token_override, ct.semantic_token)
ORDER BY ct.name;
```

Combined with the brand's pre-computed `color_scale` JSON (fetched in the same D1 batch), this
produces the full resolved token map in a single batch round-trip.

## Entity Relationships

```
brands ──────────────────────────────────────────────────────────── brand_token_overrides
  id (PK)                                                              brand_id (FK)
  seed_{l,c,h}                                                         component_token (FK)
  font_{display,heading,body}                                          semantic_token_override (FK)
  color_scale (JSON)
  valid_status                                            token_validation_log
                                                            brand_id (FK)

semantic_tokens ─────────────────── component_tokens         platform-level
  name (PK)                           name (PK)              no brand_id
  tier1_ref_light                     semantic_token (FK)
  tier1_ref_dark
  primitive_ref
```

## State Transitions: `brands.valid_status`

```
[registration] → "pending"
       ↓ (async validation via ctx.waitUntil)
   "valid"  ←─────────────────────── "invalid"
     ↑                                   ↑
(brand field update → back to pending → re-validate)
```

A brand in `pending` or `invalid` status CAN be registered and updated, but its token set MUST NOT
be served via the resolution endpoint. The resolution endpoint returns a 422 with a link to the
validation report for `invalid` brands.

## KV Value Schemas

### `brand:{id}:tokens:{scheme}` — Resolved token output (hot path)

```json
{
  "brandId": "acme",
  "scheme": "light",
  "generatedAt": "2026-03-02T12:00:00Z",
  "css": ":root { --sl-color-primary-50: oklch(0.98 0.04 240); ... }",
  "tokens": {
    "button.background.default": "oklch(0.62 0.18 240)",
    "color.surface.default": "oklch(0.98 0.01 240)",
    "font.family.body": "\"Brand Sans\", sans-serif"
  }
}
```

Both `css` (for direct `<link>` serving) and `tokens` (for JSON API) are stored together to avoid
double KV reads.
